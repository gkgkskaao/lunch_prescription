## 2. 기술 스택 / 아키텍처

- **마크업**: 시맨틱 HTML5, `lang="ko"`, 모바일 메타뷰포트, `theme-color` 지정.
- **스타일링**: Tailwind(CDN) + 인라인 설정 + 약간의 커스텀 CSS.
  - Tailwind config를 `<script>`로 런타임 확장 — 커스텀 컬러 팔레트 `rx.*`(처방전 그린 계열)와 `font-sans`(고딕)·`font-serif`(바탕) 등록.
  - 커스텀 CSS는 Tailwind로 표현이 어려운 **처방전 디테일**에 집중: 절취선(점선+양쪽 원형 노치), 바코드, 스피너, fade-up 애니메이션, 토스트.
  - 접근성 배려: `prefers-reduced-motion` 시 애니메이션 사실상 정지.
- **로직**: 순수 바닐라 JS(프레임워크 없음). 전역 상태 변수 + DOM 직접 조작.
- **상태 관리 방식**: 화면 전환은 4개 `<section>`에 `.hidden` 클래스를 토글하는 SPA식 단일 페이지 전환(라우터 없음).

## 3. 화면 구조 (4개 섹션)

`#app-header`(고정 헤더) 아래 최대폭 480px 컨테이너 안에서 4개 화면이 교체된다.

1. **`#landing-screen` 랜딩** — 슬로건("점심 뭐 먹지, 또 30분 날렸죠?"), CTA 버튼 `#start-btn`("3초 만에 처방받기").
2. **`#quiz-screen` 진단 문진** — 진행도 표시(`1/3` 텍스트 + 3개 점 인디케이터), 질문 `#question-text`, 동적 생성 선택지 `#options`.
3. **`#loading-screen` 조제 중** — 스피너 + 로테이션되는 로딩 문구 `#loading-msg`. 2.5초 고정 대기.
4. **`#result-screen` 결과(처방전 카드)** — 약봉지 카드(워터마크·℞ 기호·메뉴명·메시지·복약정보표·바코드) + 공유 버튼 2개 + 다시하기.

화면 전환은 `show(screenId)` 함수가 담당 ([index.html:322](index.html#L322)): 4개 ID를 순회하며 대상이 아니면 `hidden` 추가·`flex` 제거, 대상이면 그 반대.

## 4. 동작 흐름 (User Flow)

```
랜딩 → (start-btn) → 퀴즈 Q1 → Q2 → Q3 → 로딩(2.5초) → 결과 카드
                                                          ↓
                                          공유(카톡/인스타) · 다시하기
```

- `startQuiz()` [index.html:331](index.html#L331): 상태 초기화 후 퀴즈 진입, `showQuestion(0)`.
- `showQuestion(index)` [index.html:338](index.html#L338): 질문/진행도/점 인디케이터 갱신, 선택지 버튼을 **동적으로 생성**하고 클릭 핸들러 부착.
- `selectOption(tags)` [index.html:364](index.html#L364): 선택지의 태그를 `selectedTags`에 누적, 인덱스 증가. 마지막 문항이면 로딩→결과로 진행.
- `showLoading(callback)` [index.html:374](index.html#L374): 로딩 화면 표시, 700ms마다 문구 교체, **2500ms 후 콜백 실행**.
- `showResult()` [index.html:407](index.html#L407): 매칭 결과를 카드 DOM에 채우고 결과 화면 표시.

## 5. 데이터 모델 (`DATA` 객체, [index.html:239](index.html#L239))

### 5.1 퀴즈 (`DATA.quiz`) — 3문항

| 문항 | 질문 | 선택지 → 태그 |
|------|------|----------------|
| q1 (맛) | 오늘, 혀가 원하는 맛은? | 슴슴하게→`mild` / 맵게→`spicy` / 기름지게→`greasy` |
| q2 (멘탈) | 지금 당신의 멘탈 상태는? | 번아웃→`tired` / 빡침→`angry` / 버티는중→`survive` |
| q3 (예산) | 점심 예산, 솔직하게 | 거지→`broke` / 좀 쓴다→`rich` |

- q1·q2는 3지선다, **q3는 2지선다**. 각 선택지는 태그 1개씩만 부여.

### 5.2 결과 (`DATA.results`) — 4개

| 메뉴 | 태그 조합 | 특징 |
|------|-----------|------|
| 평양냉면 | `mild, tired, broke` | |
| 매운 갈비찜 | `spicy, angry, rich` | |
| 치즈 돈까스 | `greasy, survive, rich` | |
| 김치찌개 | `[]` (`fallback: true`) | 매칭 실패 시 기본값 |

각 결과는 `menu`(메뉴명), `message`(B급 츤데레 카피), `dosage`(복용법), `sideEffect`(예상 부작용), `combo`(함께 복용), `share`(공유 문구)를 가진다.

### 5.3 로딩 문구 (`LOADING_MESSAGES`)
3개 문구를 700ms 간격으로 순환("처방전 조제 중..." 등).

## 6. 매칭 엔진 (`getResult`, [index.html:389](index.html#L389)) — 핵심 로직

```
1. fallback 결과와 일반 후보를 분리
2. 각 후보에 대해: 후보의 태그 중 selectedTags에 포함된 개수 = score (교집합 크기)
3. 최고 점수 후보를 선택. 동점이고 score>0이면 후보 리스트에 누적
4. bestScore===0 이면 fallback 반환
5. 동점 다수면 무작위(random) 1개 선택
```

**중요한 동작 특성 (정확히 짚어둘 점):**

- **모든 태그가 동등 가중치**다. 맛(q1)·멘탈(q2)·예산(q3)을 구분하지 않고 단순 교집합 개수로만 점수를 낸다.
- 그 결과, **선택한 "맛"이 추천 메뉴와 어긋날 수 있다.** 예: `mild`(슴슴) + `angry`(빡침) + `rich`(돈씀) → 평양냉면(mild=1) vs 매운갈비찜(angry+rich=2) → **매운 갈비찜**이 나온다. 설계도 §3은 "순한맛→평양냉면" 식으로 맛을 1차 기준처럼 명시했지만, 실제 코드는 멘탈+예산 조합이 맛을 이길 수 있다.
- **3개 동점 → 랜덤** 사례 존재. 예: `greasy + angry + broke` → 세 메뉴 모두 score 1 → 무작위 추천.

## 7. 발견된 이슈 / 주목할 점

1. **fallback(김치찌개)이 사실상 도달 불가능한 죽은 코드.**
   q1의 3개 선택지(`mild`/`spicy`/`greasy`)가 각각 한 결과의 태그에 반드시 들어 있으므로, 어떤 경로든 최소 점수가 1 이상이다. 따라서 `bestScore===0` 분기(=fallback)는 정상 플로우에서 절대 발생하지 않는다. 김치찌개는 매칭으로는 노출되지 않는다.

2. **North Star(공유율)를 측정하는 코드가 없다.**
   설계도가 핵심 지표를 "결과 페이지 공유율"로 정의했지만, 공유 버튼 클릭/노출에 대한 **애널리틱스·이벤트 트래킹이 전혀 없다.** 핵심 지표가 코드상으로는 수집되지 않는다.

3. **공유 기능이 프로토타입 수준.**
   - 카톡 공유(`#share-kakao`)는 실제 카카오 SDK가 아니라 **공유 문구를 클립보드에 복사**(`navigator.clipboard`)하고 토스트 안내. 코드 주석에 "상용화 시 `Kakao.Share.sendDefault`" TODO 명시 ([index.html:449](index.html#L449)).
   - 인스타 공유(`#share-insta`)는 **토스트 안내만**("화면 캡처해서 올려보세요"). 실제 공유/이미지 생성 없음.
   - 처방전 카드를 **이미지로 저장/캡처하는 기능이 없다**(공유 바이럴이 목표인데 카드 다운로드 부재).

4. **맛 가중치 미반영** (§6 참고) — 설계 의도와 구현 로직의 미묘한 불일치.

5. **데이터가 인라인 하드코딩.** 코드 주석이 "상용화 시 `/data` API로 분리", "`quiz.js` 로직 동일"이라 적어 둠 — 현재는 모듈 분리 전 단계.

6. **결과 다양성 한계.** 일반 결과가 3개뿐이고 각 메뉴가 특정 태그 조합에 강하게 묶여 있어, 입력 조합 대비 결과 폭이 좁다.
